# 業種別デモ設計

## 目的
フロントエンドセミナーにおいて、参加者に「自分ごと化」してもらうための業種別デモを準備する。
「こんなことできるんだ、でも今すぐは自分ではできない」という驚きと危機感を与え、契約率を向上させる。

---

## 優先カバー業種（交流会で多い業種）

1. 士業（税理士、会計士、弁護士、社労士など）
2. 建設業
3. 介護
4. 広告代理店
5. 人材業界

---

## 業種別：課題とデモ案

### 1. 士業

**共通の課題：情報の厳密さ・正確性の担保**

| 細分類 | 具体的な課題 | デモ案 |
|--------|-------------|--------|
| 会計士・税理士 | 数値のズレ、計算ミスのチェックに時間がかかる | 数値の自動チェック・整合性確認システム |
| 弁護士 | 判例や法令の引用が正しいか確認が必要 | 判例・法令の自動ファクトチェック機能 |
| 社労士 | 法改正への対応、届出書類の正確性 | 最新法令に基づく書類チェック機能 |

**刺さるポイント：**
- 「この判例、本当に正しいですか？」と聞かれた時に即答できる
- ダブルチェックの時間を大幅削減
- ミスによる信用失墜リスクの回避

### 2. 建設業

**共通の課題：見積もり作成の煩雑さ、現場管理の負担**

| 課題 | 詳細 | デモ案 |
|------|------|--------|
| 見積もり作成 | 複数業種からの見積もりを集約、細かい数量計算が必要 | PDF/FAXの見積もりを一瞬で転記・集計するシステム |
| 現場での即見積もり | 現場で概算を出したいが、事務所に戻らないとできない | モバイルで即見積もり作成できる仕組み |
| 現場管理 | 進捗管理、安全管理、日報作成の負担が大きい | 現場管理の自動化・効率化システム |

**刺さるポイント：**
- 「FAXで届いた見積もり、手打ちで転記していませんか？」
- 現場から離れずに見積もり回答できる
- 「会社の見積もりテンプレートを変えずに自動化できる」という安心感

**デモの現状と準備タスク：**

| 項目 | 状態 | 備考 |
|------|------|------|
| PDF/FAX→見積もり転記システム | **ほぼ完成** | PDFを読ませると原価表と見積もりに転記される仕組みは動作中 |
| 見積もりテンプレート（Excel） | **未完成** | デザインを気合い入れて作り込む必要あり。説得力のある見た目が重要 |
| 現場で即見積もり機能 | 構想段階 | 将来的に実装予定 |

**重要ポイント：**
- 会社独自の見積もりテンプレートを変えたくないというニーズがある
- 複雑なデザインにも対応できることを見せることが重要
- 「今のやり方を変えなくていい＋自動化される」の両方を伝える

**期限：2週間以内にデモ完成（2025年2月上旬目標）**

#### 追加デモ案：「育てるナレッジAI」

**コンセプト：自社専用のAIアドバイザーを、自分たちで育てていく**

**背景・課題：**
- 建設業は人が定着しにくく、教育コストが非常に高い
- 多種多様な工事があり、知識がなければ判断ができない
- 「どの部材にどんな施工をすべきか」「この状況での選択肢は何か」「誰に聞けばいいのか」がわからない
- 専門のようで専門じゃない＝幅広い知識が必要だが、一人で全部は覚えきれない

**ソリューション：**
- 自社独自のナレッジを蓄積し、自社の工事に関してはそのAIに聞けば答えが出てくる「自社専用AIアドバイザー」
- ナレッジをどんどん追加していくことで、自分たちで育てていく感覚

**3つの強み：**
1. **誰でもアップデートできる** — ナレッジ追加の簡単さが最大のポイント
2. **誰でも質問できる** — 新人でもベテランでも同じ品質の回答を得られる
3. **正確に答えられる** — 自社のナレッジに基づいた確実な回答

**刺さるポイント：**
- 「新人が入るたびに同じこと教えていませんか？」
- 「ベテランが辞めたら、その知識ごと消えていませんか？」
- 「育てれば育てるほど、会社の資産になるAIです」

**検討事項：**
- ナレッジ追加のUI/UX（誰でも簡単にできる仕組み）
- 回答精度の担保方法（RAG等の技術選定）
- デモで見せるシナリオの具体化

**技術設計メモ：**

コア技術：RAG（検索拡張生成）
- 登録時：ドキュメントをチャンク分割 → ベクトル化 → DBに保存
- 質問時：質問文をベクトル化 → 類似検索 → LLMに渡して回答生成
- 自社データだけに基づいた正確な回答を実現

技術スタック案：
| レイヤー | 選択肢 | 備考 |
|---------|--------|------|
| フロントエンド | Next.js + Tailwind | チャットUI＋管理画面 |
| ナレッジ登録 | PDF・Word・Excel・画像・音声対応 | ドラッグ&ドロップで登録 |
| ベクトルDB | Supabase pgvector | 安価で管理が楽 |
| LLM | Gemini or Claude API | 日本語性能が高い |
| ファイル保存 | Supabase Storage | アップロード原本の保管 |
| 認証 | NextAuth | 社内メンバーのみアクセス |

ナレッジ入力手段（複数用意して差別化）：
| 方法 | ユースケース | 難易度 |
|------|-------------|--------|
| ファイルアップロード | 既存の施工マニュアル・仕様書PDF | 低 |
| テキスト入力 | 「この場合はこうする」をフォームで直接記入 | 低 |
| 音声入力 | 現場でスマホに話す → 文字起こし → ナレッジ化 | 中 |
| Q&A登録 | よくある質問と答えをペアで登録 | 低 |
| 写真+メモ | 現場写真を撮って一言メモ → ナレッジ化 | 中 |

デモMVP（最小3画面）：
1. **ナレッジ登録画面** — PDFをドラッグ&ドロップ → 「登録完了」
2. **チャット画面** — 「ALCの施工方法は？」→ 自社マニュアルに基づいた回答
3. **ナレッジ一覧画面** — 登録済みドキュメント一覧、知識件数の可視化

スライド用コピー案：
- ヘッドライン：「ベテランの知識、辞めても残る。聞けば答える自社専用AI。」
- 4特徴（2x2グリッド）：誰でもナレッジ追加 / 誰でも質問できる / 自社データで正確回答 / 育てるほど賢くなる

### 3. 介護

**共通の課題：記録業務の負担、多言語対応、レガシーシステムとの連携**

| 課題 | 詳細 | デモ案 |
|------|------|--------|
| 介護記録の転記 | 紙ベースの記録をデジタル化する手間 | 紙→デジタル自動変換システム |
| 音声入力対応 | 手が離せない現場で記録を取りたい | 音声入力→所定フォーマットへ自動整形 |
| 外国人スタッフ対応 | 母国語で書いた記録を日本語に直す必要 | 介護文脈に特化した多言語→日本語変換 |
| レガシーシステム連携 | 古い独自システムへの転記作業が煩雑 | 会計・独自システムへの転記効率化 |

**刺さるポイント：**
- 「記録のために残業していませんか？」
- 利用者と向き合う時間を増やせる
- 外国人スタッフの戦力化が加速
- 「古いシステムを変えなくても効率化できる」

**デモの現状：**

| 項目 | 状態 | 備考 |
|------|------|------|
| ほのぼのNext連携（CSV→自動転記） | **1週間以内にデモ予定** | CSVをGoogleドライブに入れると自動で転記・処理が完了する仕組み |
| 音声入力→介護記録変換 | **構想段階** | セキュリティ課題の解決が先 |

**ほのぼのNext連携デモの詳細：**
- ほのぼのNextから吐き出したCSV → Googleドライブに投入 → 自動で転記処理
- 完全な「自動化」というには遠いが、デモとしては十分
- 同じソフトを使っている介護事業者には刺さる
- このアプローチを深掘りすることで、他のシステムにも応用できる可能性あり

---

### 音声入力のセキュリティ課題（要解決）

介護現場での音声入力を実現するために、以下3つの課題を解決する必要がある。

**課題1：保存の問題**
- 個人的なツール（スマホの録音アプリ等）に保存されるとセキュリティインシデントのリスク
- **対策案：**
  - 施設側の使用規定を策定
  - 決まった部屋で使用するルール
  - 決まった機器（施設管理のデバイス）のみ使用
  - アナログな管理体制の構築が必要

**課題2：入所者への同意取得**
- 録音に対する入所者（またはその家族）からの事前同意が必要
- 新たな仕組み・フローの導入が必要
- 施設側に心理的抵抗がある可能性

**課題3：文字起こしツールの選定とデータ処理**
- 学習されないモデルを使用することが前提
- 個人情報や含まれてはならない情報が混入する可能性
- **対策案：**
  - 文字起こしデータをそのまま使わない
  - Pythonでマスキング処理を行い、個人情報を除去してから使用

**今後の方針：**
- 音声系は上記3課題の解決後に着手
- まずは「ほのぼのNext連携」でデモを実施し、実績を作る

### 4. 広告代理店

**共通の課題：AIの出力品質が低く、結局人の手が必要になる**

**背景：**
- LPやWebページをAIで作ろうとしたが「AI感」が消えない
- 結局人が作り直す必要があり、AIは使えないと思い込んでいる
- → 逆に「ちゃんとやればここまでできる」と見せれば驚きは大きい

| 課題 | 詳細 | デモ案 |
|------|------|--------|
| LP/Webページ制作 | AIで作ると品質が低い、AI感が残る | 人が作るレベルのLP/Webページを一発で生成（要：マーケ知識の組み込み） |
| 業界リサーチ | クライアントの関連業界調査に時間がかかる | 業界リサーチの自動化・一瞬で完了する仕組み |

**刺さるポイント：**
- 「AIでLP作ったけど微妙だった、という経験ありませんか？」
- 「それ、プロンプトの問題かもしれません。ちゃんとやるとここまでできます」
- 競合より早くクライアントに提案できる

**準備の課題：**
- マーケティングの専門知識を組み込んだプロンプト/システム設計が必要
- 今すぐデモを作るのは難しい → Phase 3で対応

※マーケティング知識の深掘りが必要。外部専門家との連携も検討

### 5. 人材業界

**共通の課題：書類処理・マッチング業務の効率化**

| 課題 | 詳細 | デモ案 |
|------|------|--------|
| 履歴書チェック | 大量の履歴書を目視確認している | 履歴書アップロード→自動チェック・要点抽出 |
| 面接判定 | 面接の評価基準がバラつく | 面接の自動判定フロー（評価基準の標準化） |
| 職務経歴書添削 | 求職者への添削フィードバックに時間がかかる | 職務経歴書の自動添削・改善提案 |
| 求人マッチング | 最適な求人を探すのに時間がかかる | 履歴書・経歴書・面談内容から最適求人を自動レコメンド |

**刺さるポイント：**
- 「履歴書100通、全部ちゃんと見れていますか？」
- マッチング精度向上で成約率アップ
- 求職者へのレスポンス速度向上で競合に勝つ

## セミナーでの使い方

### 事前準備
1. 参加者の業種を事前にヒアリング（申込時 or 前日確認）
2. 該当業種のデモを準備しておく

### セミナー内での流れ
1. ヒアリング：「普段どんな業務に時間がかかっていますか？」
2. 共感：「そうですよね、○○業界の方はみんなそうおっしゃいます」
3. デモ：「ちょっとこれ見てください」（2〜3分）
4. 気づき：「これ、御社でもすぐ使えますよね？」
5. ギャップ：「ただ、これを一人で作ろうとすると...」

### デモ後のクロージングへの接続
- 「このレベルのことを、自分で調べながら作るのに何ヶ月かかると思いますか？」
- 「私と一緒にやれば、1〜2ヶ月で動くものができます」
- 「しかも、作って終わりじゃなく、運用も改善も一緒にやります」
